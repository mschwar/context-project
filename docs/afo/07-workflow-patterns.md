## 2. Pre-Commit Hook Update

### 2.1 Current State

`.pre-commit-hooks.yaml` runs `ctx status . --check-exit-code`.

### 2.2 New Hook Definition

```yaml
- id: ctx-check
  name: Check ctx manifests are fresh
  entry: ctx check . --check-exit
  language: python
  pass_filenames: false
  always_run: true
```

### 2.3 Changes

| Before | After |
|--------|-------|
| `ctx status . --check-exit-code` | `ctx check . --check-exit` |

The `--check-exit` flag on the new `check` command mirrors the old `--check-exit-code` behavior: exit 1 if any manifests are stale or missing.

### 2.4 JSON Mode Hook (optional advanced pattern)

For CI pipelines that want machine-readable hook output:

```yaml
- id: ctx-check-json
  name: Check ctx manifests (JSON)
  entry: bash -c 'CTX_OUTPUT=json ctx check . --check-exit'
  language: system
  pass_filenames: false
  always_run: true
```

## 3. CI Gate Update

### 3.1 Current State

`.github/workflows/ctx-check.yml` runs:

```yaml
- name: Check CTX manifests are fresh
  run: python -m ctx status . --check-exit-code
```

### 3.2 Updated Workflow

```yaml
name: CTX Manifest Check

on:
  push:
    branches:
      - main
      - feat/**
      - fix/**
  pull_request:
    branches:
      - main
      - feat/**
      - fix/**

jobs:
  check_ctx_manifests:
    runs-on: ubuntu-latest
    steps:
      - name: Check out repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install ctx
        run: python -m pip install -e ".[dev]"

      - name: Check CTX manifests are fresh
        run: python -m ctx check . --check-exit
        env:
          CTX_OUTPUT: json
```

### 3.3 Changes

| Before | After |
|--------|-------|
| `python -m ctx status . --check-exit-code` | `python -m ctx check . --check-exit` |
| No CTX_OUTPUT | `CTX_OUTPUT: json` (optional, for parseable CI output) |

## 4. Session-End System Prompt Fragment

### 4.1 Purpose

Agents should refresh manifests at the end of a coding session. This prompt fragment can be added to system prompts or CLAUDE.md files.

### 4.2 Fragment

```markdown
## Session-End Manifest Refresh

Before ending your session, if you modified source files:

1. Run `ctx refresh .` to update manifests for changed directories.
2. Run `ctx check .` to verify all manifests are fresh.
3. If stale manifests remain, run `ctx refresh . --force` on the affected paths.

For JSON output: set `CTX_OUTPUT=json` or use `--output json`.
```

### 4.3 Where This Goes

This fragment is included in:
- The `Integration Patterns` section of AGENTS.md (see 06-docs-and-discovery.md)
- Optionally in a project's `CLAUDE.md` or equivalent

## 5. PR Context Block Workflow

### 5.1 Purpose

Automatically attach ctx manifest summaries to pull requests so reviewers (human or agent) get instant context about what directories are affected.

### 5.2 GitHub Action

Create `.github/workflows/ctx-pr-context.yml`:

```yaml
name: PR Context Block

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  add_context:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
    steps:
      - name: Check out repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install ctx
        run: python -m pip install -e .

      - name: Generate context summary
        id: ctx_export
        run: |
          export CTX_OUTPUT=json
          ctx export . --depth 1 > ctx_output.json 2>/dev/null || true
          echo "context<<EOF" >> $GITHUB_OUTPUT
          cat ctx_output.json >> $GITHUB_OUTPUT
          echo "EOF" >> $GITHUB_OUTPUT

      - name: Comment on PR
        uses: actions/github-script@v7
        with:
          script: |
            const ctx_output = `${{ steps.ctx_export.outputs.context }}`;
            let body = '## ctx Context Summary\n\n';
            try {
              const data = JSON.parse(ctx_output);
              body += '```json\n' + JSON.stringify(data.data, null, 2) + '\n```\n';
            } catch {
              body += 'ctx export not available for this PR.\n';
            }
            body += '\n---\n*Generated by [ctx](https://pypi.org/project/ctx-tool/)*';

            // Find existing ctx comment
            const comments = await github.rest.issues.listComments({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
            });
            const existing = comments.data.find(c => c.body.includes('## ctx Context Summary'));

            if (existing) {
              await github.rest.issues.updateComment({
                owner: context.repo.owner,
                repo: context.repo.repo,
                comment_id: existing.id,
                body: body,
              });
            } else {
              await github.rest.issues.createComment({
                owner: context.repo.owner,
                repo: context.repo.repo,
                issue_number: context.issue.number,
                body: body,
              });
            }
```

### 5.3 What the PR Comment Contains

The comment shows the depth-1 export: top-level directory summaries. This gives reviewers immediate context about the codebase structure without reading individual files.

## 6. Agent-to-Agent Handoff Recipe

### 6.1 First 3 Commands Pattern

When an agent is dropped into an unfamiliar codebase that has ctx installed:

```bash
# 1. Check if manifests exist and are fresh
ctx check . --output json

# 2. If stale or missing, refresh
ctx refresh . --output json

# 3. Load context for navigation
ctx export . --depth 1 --output json
```

### 6.2 Handoff Protocol

When Agent A finishes work and Agent B takes over:

**Agent A (end of session):**
```bash
# Refresh manifests to reflect changes
ctx refresh . --output json
# Verify health
ctx check . --output json
```

**Agent B (start of session):**
```bash
# Load context
ctx export . --depth 1 --output json
# Check if anything is stale (Agent A may have missed something)
ctx check . --output json
```

### 6.3 Natural Language Mapping

| Agent says... | Command |
|---------------|---------|
| "What's in this codebase?" | `ctx export . --depth 1 --output json` |
| "Are the manifests up to date?" | `ctx check . --output json` |
| "Update the documentation" / "Refresh the context" | `ctx refresh . --output json` |
| "Start fresh" / "Regenerate everything" | `ctx refresh . --force --output json` |
| "What changed?" | `ctx check . --diff --output json` |
| "Is everything healthy?" | `ctx check . --verify --output json` |
| "How much coverage?" | `ctx check . --stats --output json` |
| "Clean up" / "Remove manifests" | `ctx reset . --yes --output json` |
| "Set up ctx" / "Configure the provider" | `ctx refresh . --setup --output json` |
| "Watch for changes" | `ctx refresh . --watch` |
| "How much would this cost?" | `ctx refresh . --dry-run --output json` |

## 7. Workflow Decision Tree

For agents deciding which command to run:

```
New codebase, no CONTEXT.md files?
├── Yes → ctx refresh . --output json
│
CONTEXT.md files exist?
├── Yes → ctx check . --output json
│         ├── All fresh → ctx export . --depth 1 --output json (load context)
│         └── Stale/missing → ctx refresh . --output json
│
Modified source files during session?
├── Yes → ctx refresh . --output json (before ending session)
│
Need to understand a specific directory?
├── Yes → ctx export . --depth 2 --output json (deeper export)
│
Debugging manifest issues?
├── Yes → ctx check . --verify --output json
```
