## Problem Statement

ctx generates CONTEXT.md manifests so agents instantly know a codebase. But it's a CLI
you have to consciously run. The goal: make it feel inevitable — as automatic and
invisible as git itself. `ctx init` is the one ritual that bootstraps self-maintaining
infrastructure.

## Vision

### 10x Check
ctx init doesn't just install a git hook — it makes ctx self-maintaining infrastructure.
Multi-hook installation (post-commit, post-checkout, post-merge) ensures .ctx/ is ALWAYS
fresh, not just after commits. Depth-tiered exports let agents choose their resolution.
Hook telemetry makes the flywheel observable. Adoption tracking measures the viral loop.
The result: once you run ctx init, you never think about ctx again. It's just there.

### Platonic Ideal
You git clone any repo. If it has ctx init'd, your agent has full context before its
first prompt. The experience: the repo already knew the agent was coming.

## Key Design Decisions

### 1. .ctx/ is the export layer, not a replacement
Per-directory CONTEXT.md files stay as they are (the navigable agent map). `.ctx/` at root
is a NEW artifact: compiled exports + metadata for SDK consumers. Git hooks run
`ctx refresh` (updates per-directory manifests) then `ctx export` (populates .ctx/).

### 2. ctx init repurposes the deprecated legacy command
The legacy `ctx init` (alias for refresh) has been shipping deprecation warnings since
1.0.0. Repurposing it for the new init behavior is clean. Version bump to 1.1.0 signals
the semantic change.

### 3. Async hooks — never block the user
Git hooks run `ctx refresh` in the background and exit immediately. Users never wait.
PID file debounce prevents 10 background processes during fast rebases.

### 4. Hook invocation uses sys.executable with fallback
Hook script records the full Python path from `sys.executable` at init time, with
fallback to `python3` if the recorded path disappears (virtualenv deleted).

### 5. Hook conflict detection — chain, don't overwrite
If `.git/hooks/post-commit` already exists:
- If it has ctx marker → update in place
- If no ctx marker → append ctx block with start/end markers
- ctx uninit removes only ctx blocks, preserving pre-existing hooks

### 6. Depth-tiered exports
```
.ctx/
  CONTEXT.md          # full tree export
  CONTEXT-1.md        # depth 1 only (root + immediate children)
  CONTEXT-0.md        # root summary only
  metadata.json       # adoption tracking, version, timestamps
  hook_log.json       # hook telemetry
  .gitignore          # * (ignore everything in .ctx/)
```

### 7. metadata.json schema
```json
{
  "schema_version": 1,
  "initialized_at": "2026-03-21T22:00:00Z",
  "initialized_by": "matt",
  "ctx_version": "1.1.0",
  "repo_id": "sha256:...",
  "last_refresh": "2026-03-21T22:05:00Z",
  "python_executable": "/home/matt/.venv/bin/python",
  "hooks_installed": ["post-commit", "post-checkout", "post-merge"]
}
```

## Scope

### Accepted
- Multi-hook installation: post-commit, post-checkout, post-merge
- Depth-tiered exports: CONTEXT.md (full), CONTEXT-1.md (depth 1), CONTEXT-0.md (root)
- Adoption tracking: initialized_at, initialized_by, ctx_version, repo_id in metadata.json
- Hook conflict detection: detect existing hooks, chain (append) or warn
- ctx uninit: reverse of init, removes hooks + .ctx/ cleanly
- Hook telemetry: .ctx/hook_log.json with timestamp, trigger, duration, success/failure
- .ctx/.gitignore + root .gitignore update for .ctx/
- PID file debounce for hook dedup during fast rebases
- Version bump to 1.1.0
